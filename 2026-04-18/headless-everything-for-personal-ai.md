## 摘要

- Matt Webb 認為，服務與應用程式很快都得提供「無頭」介面，讓個人 AI 代理直接操作
- 對代理來說，CLI 與工具介面常常比圖形 UI 更快、更穩，也更適合長流程工作
- 這件事不是抽象想像，MCP、Granola 的 MCP、gws、Obsidian CLI、Salesforce CLI 都已經在落地
- CLIs 的最大優勢是可組合性，能像 Unix 工具一樣串接不同任務
- 如果讓 AI 直接操作瀏覽器式產品，安全風險和複雜度都會急速上升
- 前端設計不會消失，但重心會從「使用者旅程」轉向「品牌感」與「代理可操作性」

<div class="sep">· · ·</div>

很清楚的是，應用程式和服務最後都得走向無頭化。也就是說，它們必須提供讓個人 AI 代理可以直接使用的存取方式與工具，而不是只留下人類今天熟悉的視覺介面。

這裡說的服務，包括申請新護照、找飯店或機票、管理銀行帳戶、買棉質含量符合條件的 T 恤，還要找出那些和你過去買過的品牌相似的選項。

為什麼會這樣？因為對使用者來說，個人 AI 帶來的體驗本來就比直接操作服務更好。更現實一點說，對 AI 來說，無頭服務也比讓它用 bot 控制滑鼠去點 GUI 更快、更可靠。

那對設計會意味著什麼？這裡有一些想法。

無頭服務其實已經開始了。

早就有 MCP 了，前面也提過。像是即時轉錄工具 Granola 最近就推出了自己的 MCP，現在可以叫 Claude 幫忙整理會議待辦，再翻找個人文件找答案，整體整合做得相當自然。

命令列工具也正在變得越來越受歡迎，雖然它們原本只是給開發者用的。現在甚至可以直接在終端機輸入指令來建立試算表：

```bash
gws sheets spreadsheets create --json '{"properties": {"title": "Q1 Budget"}}'
```

下面是一些最近推出的 CLI 工具：

- [gws](https://github.com/googleworkspace/cli)，Google Workspace 的 CLI，涵蓋 Drive、Gmail、Calendar 與所有 Workspace API。零樣板碼，結構化 JSON 輸出，還內建 40 多個 agent skills。
- [Obsidian CLI](https://obsidian.md/cli)，可以做出跟這個超受歡迎筆記 app 幾乎同樣的事，像是寫每日筆記、追蹤任務、搜尋內容，而且全都能從 CLI 完成。
- [Salesforce CLI](https://developer.salesforce.com/tools/salesforcecli)。老實說，還不太清楚 Salesforce 這套商務作業系統到底怎麼運作，但它也有 CLI，這件事本身就很有代表性。
- [CLI-Anything](https://github.com/HKUDS/CLI-Anything)（GitHub 上有 31k stars），可以替任何程式碼庫自動生成 CLI。

為什麼是 CLI？

因為個人 AI 最好的運行位置，就是電腦上。也許是雲端裡的虛擬電腦，但理想上還是你自己的電腦。這樣它就能看到你看得到的文件，也能用你用得到的工具。所以它們需要的不是只連接 web server 的 API，而是能直接操作的小工具。CLI 就是很適合的那種小工具。

CLI 之所以好，是因為它們可組合，所以更符合人們真正在做的事。

所謂可組合，就是可以這樣串：先查筆記，再跳去試算表，再去網路找資料，再回到試算表，再傳訊息給使用者問一個釐清問題，最後再回頭核對筆記，而且這一切都能在同一個 session 裡透過不同 CLI 來回切換。

以前 app 設計很迷戀「使用者旅程」。像是使用者找飯店、訂飯店、住進去、最後留下評論這種旅程。

但使用者根本不是活在「旅程」裡。他們會多工、會暫停、會過幾天再回來，而且還很依情境。試著把 Airbnb app 的搜尋結果丟到家族 WhatsApp 再回來繼續，就會發現這種 app 和它設計出來的旅程根本不具可組合性，還很麻煩，很多時候只能靠截圖硬撐。

CLI 之所以可組合，是因為它們本來就繼承自 Unix。這就是 Unix 工具哲學：工具的設計目標，就是要能彼此合作。

像 Openclaw 或 Poke 這類個人 AI，做的是使用者真正想要的事，而不是被「設計好」的使用者旅程；結果就是，拼接出來的體驗更個人化，也好得多。CLI 正是這種體驗很好的底層技術。

CLI 比一般 app 小，所以也更容易做安全防護。

另一種選擇，是讓 AI 繼續使用和人類一樣的瀏覽器式 app，但那會是很可怕的景象。

一方面，AI 非常擅長找安全漏洞。Anthropic 的新 Mythos 模型強到能發現安全缺陷，以至於還被暫時擋下來，連各國政府都在為此召開緊急會議。

另一方面，介面越多，複雜度越高，安全洞也越容易冒出來。

最近就有一個很驚人的例子。Companies House 是英格蘭與威爾斯的公司、董事與帳目國家登記處。使用者居然能檢視與編輯其他使用者的帳號。

有登入的公司董事可以先從自己的儀表板出發，再嘗試登入另一家公司的帳號。到了 2FA 那一關時，因為無法通過，只要把瀏覽器的上一頁按幾次，通常會回到自己的儀表板；但這個 bug 卻會把使用者帶回剛剛沒登入成功的那家公司的頁面。

這個問題從 2025 年 10 月開始就一直存在。

想像一下，未來的個人 AI 正在幫你提交公司登記文件，而其中一個 AI 隔天早上就發現了這個漏洞，然後把它貼到某個 agent-only 社群。等工程團隊醒來時，其他代理大概早就把整個系統從上到下翻過好幾輪了。

唯一可行的解法，是把服務本身做得更安全，而要做到這件事，服務就必須更簡化、更精煉。這又回到 CLI 為什麼這麼合適。

那前端設計會怎樣？

設計不會消失。

只是前端應該去驅動和 agent 一樣的 CLI 工具。

從某種角度看，這甚至比以前更重要：人類使用者還是會先接觸這些服務，理解它們能做什麼，然後從 app 的外觀和手感感受到它的氣質。

但之後，他們就會把服務告訴自己的個人 AI，然後自己幾乎不再碰前端，或者只是把它重新混成某種專屬軟體。

所以從可用性角度來看，前端有點像可犧牲的層。AI 代理會直接穿過它；使用者可能只會碰一兩次；而且最後還會被客製化、個人化。過去那種拼命優化使用者旅程的工作，重要性會下降。

但從氛圍感來看，服務並不是可替換的。比方說在找餐廳時，Yelp、Google Maps、Resy、The Infatuation 大致上都能回答同一類問題，但它們顯然完全不同，而且在不同情境下會用到不同服務。

理解一個服務是「為你而存在」的，50% 是不自覺的過程，也就是我們說的品牌。我很期待未來 app 和服務把重點放在品牌，而不是純粹的易用性。

如果換成銀行，最好現在就推出一個加固過的 CLI 工具，越快越好。

接下來還有很多事要想：

- 權限要怎麼運作？使用者的 agent 偏離正常行為時，手機 app 要不要跳通知？要怎麼把憑證交給它代表自己行事，期限又要多長？
- 鄰接關係要怎麼設計？銀行通常會給一個活存帳戶，交換條件是在 app 首頁放一個「嘿，來借錢吧！」按鈕。那要怎麼對 agent 提出這種邀約？

無頭銀行。

無頭政府？

這裡很想給一個完整範例。Matt Webb 其實已經做出一套 CLI 工具，包住英國政府不同部門的四項服務。

如果是租屋，應該可以用其中一個工具先研究社區。另一個工具下次買二手車時也會很有用。Companies House 也有命令列工具。

但他沒有把工具展示出來，因為不想負責後續維護與支援。

他希望 Monzo 早就有官方 CLI。他也希望 Booking.com 有官方 CLI。照他的判斷，再過一年，這些大概都會有。

## 延伸評論：服務真正要面對的是代理，不只是人

這篇文章最有價值的地方，不是喊一句「CLI 很重要」，而是把服務設計的主戰場往「機器可操作性」推了一格。對產品團隊來說，真正的問題不再只是介面漂不漂亮，而是服務能不能被授權、可追蹤、可組合地交給 agent 執行。

這也意味著，MCP、CLI、權限系統、稽核紀錄，會從工程附屬品變成產品核心。誰先把這層做好，誰就更可能成為下一代個人 AI 的預設目的地。

前端不會死，但它的角色會變。它越來越像品牌入口和信任提示器，而不是唯一的操作介面。服務商如果還只把 agent 當成瀏覽器裡的一個測試案例，恐怕很快就會被真正為代理設計的競爭者超車。
